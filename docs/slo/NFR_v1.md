## Reliability / Incidents

**MTBF** Initial — 1.800ч — раз в 75 дней допустим сбой.

**Errors rate** 10.000 errors/day (ошибки сервера — 5xx)

**Failure rate**

```
failure rate = (errors/day / requests/day) * 100%
             = (10.000 / (1.000.000 users * 50 requests/user)) * 100%
             = (10.000 / 50.000.000) * 100%
             = 0.02%
```

## Availability

```
A = MTBF / (MTBF + MTTR) = 1.800 / (1.800 + 1) = 99.9%
```

**Почему не через uptime/downtime**

На данный момент формула `A = (Uptime - Downtime) / Uptime` не учитывается, т.к сервис еще не запущен, а формула через MTBF/MTTR позволяет поставить более явные и понятные рамки качества и продолжительности работы сервиса.

**Провалы (3 девятки)**

Время, включая задержки провайдера не включенные в SLA:

- неделя — 12 мин
- месяц — 50 мин
- год — 9 часов

**Целевые показатели**

- 4 девятки через увеличение MTBF

## BOTEC

### Характеристики

- rate-limit/h/user: 10.000 запросов
- rate-limit-max = 1.000.000 × 10.000 / 3600 = 2.777.777 RPS теоретический максимум
- Window-peak (пик работы пользователей): 50% (12 часов активных)
- DAU = 1.000.000

### RPS

```
Пользователи:
RPS_users = 1.000.000 * 50 / 86400 = 578 RPS

Боты и автоматизация (оценочно ~3% DAU используют full rate-limit):
RPS_bots ≈ 1.900 RPS

Итого (консервативная оценка):
RPS_AVG  = ~2.500 RPS
RPS_PEAK = *3    = 7.500 RPS
headroom = +30%  = 9.750 RPS
```

### Latency (грубые оценки до реальных замеров)

```
Погрешность +20мс = GC паузы JVM (~5-10мс) + network overhead (~5мс) + queue waiting (~5мс)

latencyAvg(50%/70%)   = p50(50%/70%)  + 20мс
latencyHigh(90%/100%) = p50(90%/100%) + 20мс
```

### Concurrency (закон Литтла: C = latency_сек * RPS)

```
Concurrency(50%/70%)   = 0.15с * 2.500 = 375  concurrent requests
Concurrency(90%/100%)  = 0.30с * 7.500 = 2.250 concurrent requests
```

### Трафик

```
request-size  = 1 Кб   (входящий запрос — GET/POST с параметрами)
response-size = 25 Кб  (исходящий ответ — JSON с задачами/данными)

ingress = 7.500 * 1 Кб  = 7 мб/c   = 0.006 Gbps  (входящий)
egress  = 7.500 * 25 Кб = 183 мб/c = 0.17 Gbps   (исходящий)
```

## Latency

p99.9 — пока не таргетируется.

Latency через гистограммы пока не берем — нет реальной нагрузки и готовых инстансов для замеров. Пересмотреть после первого нагрузочного теста.

### edge / gateway

**Нагрузка 50%/70% (окно 50% на активный день)**

```
p50 = 150 мс  (2.350 из 2.500 RPS — основная масса)
p95 = 280 мс  (125 RPS)
p99 = 390 мс  (25 RPS)
```

```
latency_avg = (p50 * 2350 + p95 * 125 + p99 * 25) / 2500
            = (150 * 2350 + 280 * 125 + 390 * 25) / 2500
            = (352.500 + 35.000 + 9.750) / 2500
            = 160 мс
```

**Нагрузка 90%/100%**

Добавляем погрешность: GC паузы, network overhead, queue waiting (+30-40мс)

```
p50 = 250 + (30-40) мс  (2.350 из 7.500 RPS — основная масса)
p95 = 350 + (30-40) мс  (125 RPS)
p99 = 450 + (30-40) мс  (25 RPS)
```

```
latency_avg = (p50 * 2350 + p95 * 125 + p99 * 25) / 2500
            = (290 * 2350 + 390 * 125 + 490 * 25) / 2500
            = (681.500 + 48.750 + 12.250) / 2500
            = 297 мс ≈ 270-300 мс
```

## Scalability

Количество пользователей будет многократно увеличиваться.

**Шаг 1 — RPS_PEAK превышен на 10-15%**

Увеличиваем read replica БД на +1. Если серверы всё равно проседают — добавляем ещё +1.

**Шаг 2 — нагрузка остаётся после шага 1**

Увеличиваем Redis cluster на +1 (по той же схеме что и БД реплики: +1, смотрим, если не хватает — +2). Если сервер справляется — увеличиваем rate-limit на 10%. Если не справляется — переходим к шагу 3. Обязательный запас мощности: 20-30%.

**Шаг 3 — RPS_PEAK превышен на 20%+**

Scale-up инстансов. Если сервер выдерживает — rate_limit +10%, после нагрузочного теста ещё +15%. Параллельно рассматриваем scale-out на доп. инстансы (+1, смотрим результат, при необходимости +2).

**Шаг 4 — после стабилизации нового RPS**

Добавление очередей для сглаживания пиков и повышения отказоустойчивости:

- Запас > 30% после масштабирования → **Kafka** (надёжнее при больших объёмах, replay сообщений, более гибкая настройка, но выше потребление ресурсов и latency)
- Запас < 30% → **RabbitMQ** (быстрее при малых объёмах, проще в настройке, меньше overhead)
