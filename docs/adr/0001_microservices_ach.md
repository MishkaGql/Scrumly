# ADR-0001: Microservices вместо монолита/модульного монолита

- Status: #Accepted
- Date: 2026-03-02
- Owners: Backend team
---
## Context

Система предполагает активное внедрение нового функционала не только от основной команды, но и от сторонних разработчиков:
  - issues в Github
  - Собрания с аналитиками из других компаний и команд 
  - Открытый контрибьютинг

Основные требования:

  - Предпологается непрервные мастшабирование, внедрение новых идей, замена старых и удаление других.
  
  - Система должна быть отказоустойчивой,тестируемой со стороны всей системы, так и отдельных частей.

  - Система должна быть производительной, позволяющей быстро обрабатывать запросы клиентов

  - DAU 1.000.000 и >2.500 RPS (по SLO), что требует scale-up и sclae-out.

---
## Decision

Используем **Microservices-architecture**:
      1) Первоначальная цель использовать Keyсloack как источник аутентификации
      2) Используем Auth service для взаимодействия с Keycloack для тонкой настройки аутентификации  и авторизации пользователей
      3) Notification service для управление уведомлениями с Websocket и
      Redis pub/sub
      5) На данный момент scrumly servic будет представлять собой модульный монолит, который после аналаза, выделения изолированных частей, будет разбит еще на микросервисы
      6) Config server для общих настроек всех сервисов
      7)  Api Gateway (async) для прокси запросов , rate-limit, контроль latency
      8) Используем SAGA для распределенных транзакций и консистентности данных
      9) Observability через alloy и grafana, отображающая loki,traces,metrics
---
## Alternatives considered

  1) __Монолит__ - Плох для scale-out, что критично для сервисов, требующих большой производительности, нельзя деплоить независимо, scale-up рано или поздно упрётся в деньги и в ограничения по железу. Сложность внедрения, удаление, замену функций, учитывая поддержку контрибьютинга

  2) __Модульный монолит__ - вариант, который мы выбрали для Scrumly service, пока не зная какие отдельные части можно из него выделить. В остальном у модульного монолита такие же проблемы как и у монолита.
---
## Consequences

**Pros**:
  - большое пространство для scale-up, scale-out
  - возможность использования нескольких баз данных, которые можно реплицировать + шардировать, что дает еще больший прирост
  - быстрый откат определенных частей + удобное тестирование

**Cons/Risks**:
  - Время на отладку, корректную настройку SAGA
  - Увеличиение Service time latency из-за перещения запросов из сервиса к сервису
  - Увеличенные шансы на replication lag и eventual consistency в редисе

**Observability**:
    - сложность поддержки системы - требует строгой проверки деплоя, тестирования, логирования и мониторинг проблем
