# ADR-0003: Использование PostgreSQL как основную базу данных

- Status: #Accepted
- Date: 2026-05-03
- Owners: Backend team
---
## Context

Для scale-up,scale-out должны быть очень мощные и разносторонние ресурсы. 
База данных должна быть быстрой, поддерживать нормальную репликацию и шардирование, иметь стабильность, большой функционал

---
## Decision

Используем **Postgresql**:
      - первоначально настраиваем по 1 базе данных на каждый сервис
      - после нарастания кодовой базы, на основе тестов производительности
      если не хватает делаем пункт и дальше смотрим на тесты и так каждый раз:
        - партиционирование
        - репликация
        - шардирование
          
      
      - веренный временем и используемый всеми ентерпрайз гигантами, открытый исходный код
      - постоянно развивается, большое комъюнити, новые патчи,тысячи  расширений позволяет подстраивать бд под свои нужды
      - репликация + шардирование
      - многофункциональная настройка под конкретный продукт,
      - большой набор индексов(b-tree,gin,gist)
      - поддержка большого числа типов данных, включая в геопространственные
      - полная поддержка ACID с тонким контролем транзакций
---
## Alternatives considered

  1) __MySQL:
    - хуже в настройке,
    - представлены только стандартные индексы,
    - неплох для небольших веб приложений
    - производительность, поддержка хуже чем у PostgreSQL
  1) Oracle 
    Минусы:
      - ушел с российского рынка, 
      - дорогое лицензирование, 
      - ограниченная поддержка типов данных и типов индексов,
      -  закрытый исходный код, 
      - далек от SQL- стандарта, 
      - гораздо меньше специалистов 
    Плюсы:
      - есть встроенное шардирование, в потсгрес достигается за счет сложностей и расширений. 
      - Есть поддержка multi-master, когда у постгрес только 1 мастер
      - Большая производительность
---
## Consequences

**Pros**:
  - большое пространство для scale-up + scale-out(частичный) и с Postgres Pro полноценный 
  - хранение не только стандартных типов данных
  - больше типов индексов
  - полностью бесплатная и с открытым исходым кодом
  - соответвует SQL-стандарту
  - большое число специалистов
  - огромная поддержка комъюнити, открытый исходный код(возможность контрибьтить)

**Cons/Risks**:
  - scale-out все равно хуже чем у Oracle
  - нет нативного шардирования(только через расширения)(в Postgres Pro минус аннулируется)
  - плохие 2PC
  - большое протребление памяти
  - изначально отсутствует  multi-master - однако решается добавление расширения

**Observability**:
    - Требует более тщательная работа со scale-out, с построением максмально эффективной работы с бд
